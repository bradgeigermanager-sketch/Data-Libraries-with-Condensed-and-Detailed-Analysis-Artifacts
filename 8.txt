This is a highly effective expansion of the ecosystem. By establishing standardized ingestion workflows, the system transitions from a closed library into an active processing engine capable of distilling external knowledge graphs into atomic logic.
Below are the structural maps and checklists for information derivation. They are separated into **Agentic** (machine-driven, automated pipelines) and **Human** (living user, curated synthesis) workflows, starting with the generalized derivation routine.
### Part 1: Generalized Information Source Extraction Map
This routine is designed to map any unstructured data source (e.g., whitepapers, statutes, news articles, or raw datasets) into the system's strict logical architecture.
#### 1A. Agentic LLM Workflow (Automated Pipeline)
**NOTATION:** Generalized Automated Derivation Engine.
**USE:** A programmatic pipeline where an autonomous agent ingests raw text, parses it for causal or definitional relationships, and autonomously POSTs the standardized JSON map to the database receiver.
**Processing Sequence:**
 1. **Ingestion & Chunking:** Retrieve unstructured text via scraper or API. Segment into discrete thematic chunks.
 2. **Semantic Triangulation:** Analyze text for logical operators (e.g., "therefore", "requires", "is defined as", "contradicts").
 3. **Artifact Generation:** Translate semantic findings into the condensed_logic string structure using [[UUID|Text]] placeholders for entity extraction.
 4. **Edge Mapping:** Query the existing library via the /api/directory endpoint. If an entity exists, retrieve its UUID. If not, generate a new UUID and queue it for artifact creation.
 5. **Payload Compilation:** Assemble the fillable_schema JSON transit map.
 6. **Commit:** Transmit to /api/admin/ingest.
**Agentic Checklist (Execution Verifiers):**

| Phase | Verification Trigger | Target State |
| :--- | :--- | :--- |
| **Parsing** | verify_claim_isolation() | Text chunk contains at least one explicit premise and conclusion. |
| **Parsing** | verify_no_hallucination() | The generated verbose description maps exactly to the source text without extrapolation. |
| **Mapping** | verify_uuid_integrity() | All generated logical links reference valid system UUIDs or newly queued UUIDs. |
| **Formatting** | verify_json_schema() | Payload strictly adheres to the atomic logic JSON standard. |

#### 2B. Living User Workflow (Wikipedia Curation)
**NOTATION:** Wikipedia Human Curation Routine.
**USE:** Methodology for a human to utilize Wikipedia not just as an encyclopedia, but as a scaffold for mapping complex historical, scientific, or narrative ecosystems into the UI.
**Processing Sequence:**
 1. **Core Anchor Generation:** Create a primary logic node for the main Wikipedia article subject.
 2. **Premise Extraction:** Read the article's sub-sections. For every sub-section that details a distinct mechanism, rule, or historical event, create a secondary logic node.
 3. **Sub-System Linking:** Use the Admin UI to map all secondary nodes back to the primary anchor using requires or extends relationships.
 4. **Conflict Mapping:** Review the "Controversies" or "Criticism" sections of the Wikipedia article. Create logic nodes for opposing viewpoints and link them using the contradicts relationship edge.
**Living User Checklist:**
 * [ ] Have I accurately captured the primary definition from the article's lead paragraph?
 * [ ] Did I explicitly map the Wikipedia Permalink (using the permanent specific revision ID, not just the general URL) into the Foundational Data to ensure source immutability?
 * [ ] Have I converted the most critical "See Also" links into active relationship edges in the Admin UI?
 * [ ] Are competing theories or criticisms represented as distinct artifacts with a contradicts edge, rather than being mashed into the primary node's description?
